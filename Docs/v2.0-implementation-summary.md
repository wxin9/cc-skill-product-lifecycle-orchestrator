# Product-Lifecycle v2.0 实施总结

**完成日期**：2026-04-16
**版本**：v2.0.0
**架构**：脚本编排 + 交互暂停

---

## 一、架构重构概览

### Before（v1.x 模型驱动）

```
模型读 SKILL.md → 手动调用 ./lifecycle 命令 → 命令独立执行 → 模型记忆下一步
```

**核心问题**：
- 模型中途失忆，后续脚本不执行
- 用户需手动驱动每一步
- 无统一状态管理

### After（v2.0 脚本编排）

```
模型调用 ./orchestrator run --intent <intent>
  → Orchestrator 自动执行 Phase 序列
  → 遇到交互节点暂停
  → 通知模型
  → 模型处理后 resume
  → Orchestrator 继续执行
```

**核心优势**：
- Orchestrator 保证执行完成
- 无需模型记忆下一步
- 统一状态管理（Checkpoint）
- 交互暂停 + 失败恢复

## 二、核心组件

### 2.1 PHASES 定义表（`scripts/core/phases.py`）

**职责**：声明式配置所有 Phase

**数据结构**（15 字段）：
```python
class PhaseDefinition(TypedDict):
    id: str                          # Phase ID
    name: str                        # 显示名称
    description: str                 # 描述
    order: int                       # 执行顺序

    auto: bool                       # 是否自动执行
    command: Optional[str]           # 对应的命令
    command_args: Optional[dict]     # 命令参数

    depends_on: List[str]            # 前置依赖
    blocks: List[str]                # 阻塞的 Phase

    artifacts: List[PhaseArtifact]   # 产物验证规则
    validation_type: Optional[str]   # 文档验证类型

    on_failure: Literal["pause", "retry", "skip"]  # 失败策略
    max_retries: int                 # 最大重试次数

    pause_for: Optional[str]         # 暂停原因
    timeout_hint: Optional[str]      # 超时提示

    intent_triggers: List[str]       # 触发此 Phase 的意图
```

**Phase 清单**（11 个）：
1. Phase 0 — 意图识别（交互）
2. Phase 1 — 项目初始化（自动）
3. Phase 2 — AI 协作 PRD 起草（交互）
4. Phase 3 — PRD 验证 + 自动快照（自动）
5. Phase 4 — 架构访谈 + 项目类型识别（交互）
6. Phase 5 — AI 协作架构设计（交互）
7. Phase 6 — 架构验证 + ADR 注册 + 快照（自动）
8. Phase 7 — 自适应测试大纲生成（自动）
9. Phase 8 — Velocity 感知迭代规划（自动）
10. Phase 9 — 迭代执行循环（交互）
11. Phase 10 — 变更处理（自动）

**关键特性**：
- 声明式配置，易于扩展
- 依赖关系明确
- 失败策略可配置
- 交互节点标注清晰

### 2.2 Checkpoint 管理器（`scripts/core/checkpoint_manager.py`）

**职责**：Phase 级别状态持久化

**Checkpoint 文件格式**（`.lifecycle/checkpoint.json`）：
```json
{
  "version": "2.0",
  "project_name": "xxx",
  "created_at": "2026-04-16T...",
  "updated_at": "2026-04-16T...",
  "current_phase": "phase-3-validate-prd",
  "status": "in_progress" | "paused" | "completed" | "failed",
  "completed_phases": ["phase-0-intent", "phase-1-init"],
  "phase_data": {
    "phase-3-validate-prd": {
      "started_at": "...",
      "completed_at": "...",
      "score": 85
    }
  },
  "intent": "new-product",
  "user_input": "我想做一个...",
  "metadata": {}
}
```

**核心方法**：
- `load()` — 加载 checkpoint，自动迁移旧版格式
- `save(checkpoint)` — 保存 checkpoint
- `init(project_name, intent, user_input)` — 初始化新 checkpoint
- `is_phase_completed(phase_id)` — 检查 Phase 是否完成
- `record_phase_start(phase_id)` — 记录 Phase 开始
- `record_phase_complete(phase_id, data)` — 记录 Phase 完成
- `record_phase_failed(phase_id, error)` — 记录 Phase 失败
- `record_phase_paused(phase_id, reason)` — 记录 Phase 暂停

**自动迁移**：
- 从 `.lifecycle/steps/*.json` 迁移到 `checkpoint.json`
- 映射旧版 step_id 到新版 phase_id
- 保留原有完成状态

### 2.3 意图解析器（`scripts/core/intent_resolver.py`）

**职责**：用户输入 → 意图类型

**意图类型**（10 种）：
```python
IntentType = Literal[
    "bug-fix",        # Bug 修复（优先级 1）
    "gap",            # 需求遗漏（优先级 2）
    "prd-change",     # PRD 变更（优先级 3）
    "code-change",    # 代码变更（优先级 4）
    "new-feature",    # 新增功能（优先级 5）
    "arch-change",    # 架构调整（优先级 6）
    "new-iteration",  # 新迭代（优先级 7）
    "continue-iter",  # 继续迭代（优先级 8）
    "new-product"     # 新产品（优先级 9）
]
```

**匹配机制**：
- 正则匹配关键词（如 "报错"、"bug" → bug-fix）
- 优先级排序（数字越小优先级越高）
- 复合意图处理（返回多个意图，串行执行）

**示例**：
```python
resolve("修了个 bug，顺便想调整下需求")
# 返回：(["bug-fix", "prd-change"], "匹配详情...")
# 执行：先 bug-fix 流程，再 prd-change 流程
```

### 2.4 Orchestrator 编排引擎（`scripts/core/orchestrator.py`）

**职责**：自动执行 Phase 序列

**核心方法**：
- `run(intent, from_phase, user_input) → exit_code` — 主入口
- `_execute_phase(phase, checkpoint) → result` — 执行单个 Phase
- `_build_execution_path(start_phase_id, intent) → path` — 构建执行路径
- `_notify_pause(phase, reason)` — 发送暂停通知
- `_notify_failure(phase, error)` — 发送失败通知

**状态机转换**：
```
initialized → in_progress → paused → in_progress → completed
                          ↘ failed → paused → in_progress
```

**执行流程**：
1. 加载或初始化 checkpoint
2. 根据 intent 确定起始 Phase
3. 构建执行路径（Phase 序列）
4. 遍历执行每个 Phase：
   - 检查依赖
   - 记录 Phase 开始
   - 执行 Phase（命令 + 验证）
   - 处理暂停/失败
   - 记录 Phase 完成
5. 所有 Phase 完成后设置 status=completed

**通知机制**：
- 双重通知：stdout 打印 + `.lifecycle/notification.json` 文件
- 通知类型：pause_for_user / validation_failed / dod_failed / error
- 包含建议行动（actions 字段）

### 2.5 CLI 重构（`scripts/__main__.py`）

**破坏性变更**：
- ❌ 删除所有旧命令（init, validate, draft, plan, outline, gate, change 等）
- ✅ 只保留 orchestrator 子命令

**新命令**：
```bash
# 启动编排
python -m scripts orchestrator run --intent <intent> --user-input "<输入>"

# 恢复执行
python -m scripts orchestrator resume --from-phase <phase-id>

# 查看状态
python -m scripts orchestrator status

# 取消流程
python -m scripts orchestrator cancel
```

**包装脚本**（`./orchestrator`）：
```bash
#!/bin/bash
SKILL_PATH="$(cd "$(dirname "${BASH_SOURCE[0]}")" && pwd)"
export PYTHONPATH="$SKILL_PATH"
python -m scripts orchestrator "$@"
```

## 五、已知限制

### 5.1 命令执行暂时禁用

**问题**：Orchestrator 目前无法直接调用 core 模块函数

**临时方案**：跳过命令执行，只验证 artifacts

**后续计划**（v2.1）：实现直接调用 core 模块函数

### 5.2 无并行 Phase 支持

**问题**：所有 Phase 串行执行，即使无依赖关系

**后续计划**（v2.2）：支持无依赖的 Phase 并行执行

### 5.3 无条件分支支持

**问题**：所有项目走相同的 Phase 序列

**后续计划**（v2.2）：支持基于项目类型的条件分支
